## How to Report Bugs

If you've found a bug, please [open an issue](https://github.com/Manirider/Manirider/issues/new) and include the following details so it can be investigated quickly:

1. **Summary** — A clear, concise description of what the bug is.
2. **Steps to Reproduce** — A numbered list of steps to reliably reproduce the issue.
3. **Expected Behavior** — What you expected to happen.
4. **Actual Behavior** — What actually happened.
5. **Environment Details** — OS, browser, language/runtime version, or any other relevant context.
6. **Screenshots or Logs** — If applicable, attach screenshots, error messages, or stack traces.

> **Tip:** Search existing issues before opening a new one — your bug may already be reported!

## How to Request Features

Have an idea for a new feature or enhancement? I'd love to hear it!

1. [Open an issue](https://github.com/Manirider/Manirider/issues/new) with the label `enhancement`.
2. Describe the **problem** you're trying to solve, not just the solution.
3. Explain the **proposed solution** and why it would be valuable.
4. Include any **examples**, mockups, or references that help illustrate the idea.

Feature requests are discussed openly, so feel free to comment on existing requests with your thoughts and use cases.

## How to Submit Pull Requests

Ready to write some code? Here's the step-by-step process:

1. **Fork the repository**
   Click the "Fork" button at the top-right of the repository page to create your own copy.

2. **Clone your fork**
   ```bash
   git clone https://github.com/<your-username>/Manirider.git
   cd Manirider
   ```

3. **Create a feature branch**
   Use a descriptive branch name that reflects your change:
   ```bash
   git checkout -b feature/your-feature-name
   # or for bug fixes:
   git checkout -b fix/bug-description
   ```

4. **Make your changes**
   Keep changes focused and minimal. Avoid mixing unrelated changes in a single PR.

5. **Commit with a clear message**
   Follow the [Commit Message Conventions](#commit-message-conventions) described below.
   ```bash
   git commit -m "Add feature X to improve Y"
   ```

6. **Push your branch**
   ```bash
   git push origin feature/your-feature-name
   ```

7. **Open a Pull Request against `main`**
   - Go to the original repository on GitHub and click "New Pull Request".
   - Select your fork and branch as the source.
   - Fill in the PR description explaining **what** changed and **why**.
   - Link any related issues (e.g., `Closes #3`).

8. **Respond to feedback**
   Be open to review comments and make any requested changes promptly.

## Commit Message Conventions

Clear commit messages make the project history easy to understand. Please follow these guidelines:

- **Use the imperative mood** — write commands, not descriptions.
  - ✅ `Add feature X`
  - ✅ `Fix bug in Y`
  - ❌ `Added feature X`
  - ❌ `Fixing bug in Y`
- **Keep the subject line short** — aim for 50 characters or fewer.
- **Separate subject from body** — if you need more context, leave a blank line after the subject and add a longer explanation.
- **Reference issues when relevant** — e.g., `Fix layout issue in README (closes #5)`.

**Examples:**
```
Add dark mode support to profile card
Fix broken image link in README
Update dependencies to latest stable versions
Remove deprecated badge from header
```

## Code Style Guidelines

To keep the codebase clean and consistent:

- **Match the existing style** — look at how the surrounding code is written and follow the same patterns.
- **Comment meaningfully** — add comments where the *why* isn't obvious, not the *what*.
- **Keep it simple** — prefer clear, readable code over clever one-liners.
- **Format before committing** — run any project-specific linters or formatters before pushing your changes.
- **Avoid unnecessary changes** — don't reformat unrelated code or change whitespace unless it's part of your fix.

## Code of Conduct

This project is committed to providing a welcoming and inclusive environment for everyone. By participating, you agree to:

- Be **respectful and constructive** in all interactions.
- Welcome contributors of all backgrounds and experience levels.
- Accept feedback graciously and offer it kindly.
- Focus on what is best for the community and the project.

Harassment, discrimination, or any form of harmful behavior will not be tolerated. If you experience or witness unacceptable behavior, please reach out via email (see [Getting Help](#getting-help)).

This project follows the principles of the [Contributor Covenant](https://www.contributor-covenant.org/).
